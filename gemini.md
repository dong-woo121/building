# 프로젝트: Unit ID Matcher (매물 호수 식별 시스템)

네이버 부동산 매물 정보와 국토교통부 건축물대장 데이터를 교차 검증하여, 외부로 공개되지 않는 매물의 "정확한 호수"를 찾아내는 데이터 매칭 엔진입니다.

## 🎯 핵심 원칙 (Mandates)
- **정확성 최우선**: 다른 부가 기능보다 "정확히 몇 호인지"를 식별하는 로직의 신뢰도가 가장 중요합니다.
- **최소한의 UI**: 데이터 입력을 위한 Input Box와 결과 리턴을 위한 단순한 View로 구성합니다.
- **효율적 스택**: 무료 배포가 가능하고 구현이 빠른 최소한의 기술 스택을 사용합니다.

## 🛠️ 기술 스택 (Proposed)
- **Language**: Python 3.10+ (데이터 처리 및 API 호출 효율성)
- **Backend/UI**: Streamlit (가장 간단한 UI + Python 서버 통합 환경)
- **API**: 공공데이터포털 - 건축물대장 서비스 (전유부, 표제부)
- **Scraper**: Playwright (네이버 매물 정보 획득용 - 필요 시)
- **Deployment**: Cloudtype 또는 Local (무료 티어 활용)

## 🔄 매칭 알고리즘 프로세스
1. **Input Stage**: 매물 주소(단지명), 동, 층, 전용면적, 거실 방향 입력.
2. **Data Acquisition**:
   - 국토부 API를 통해 해당 주소/동/층의 모든 '호별 전용면적' 리스트 확보.
   - 네이버 부동산에서 해당 매물의 상세 정보(방향 등) 확보.
3. **Filtering Logic**:
   - **Level 1**: 동/층 필터링 (건축물대장 데이터 기반).
   - **Level 2**: 전용면적 정밀 대조 (소수점 포함 일치 여부).
   - **Level 3**: (모호성 발생 시) 거실 방향과 호수별 배치 규칙 대조.
4. **Output**: 확정된 호수 번호 또는 후보군 리스트 출력.
