# 프로젝트: Unit ID Matcher (매물 호수 식별 시스템)

네이버 부동산 매물 정보와 국토교통부 건축물대장 데이터를 교차 검증하여, 외부로 공개되지 않는 매물의 "정확한 호수"를 찾아내는 데이터 매칭 엔진입니다.

## 🎯 핵심 원칙 (Mandates)
- **정확성 최우선**: 다른 부가 기능보다 "정확히 몇 호인지"를 식별하는 로직의 신뢰도가 가장 중요합니다.
- **최소한의 UI**: 데이터 입력을 위한 Input Box와 결과 리턴을 위한 단순한 View로 구성합니다.
- **무료 API만 사용**: 유료 API 또는 유료 서비스 사용 금지.

## 🛠️ 기술 스택
- **Language**: Python 3.10+
- **Backend/UI**: Streamlit (Python 서버사이드 — 모든 로직 및 API 호출은 서버에서 실행, 브라우저 노출 없음)
- **주소 파싱**: 카카오 주소 검색 API → sigunguCd, bjdongCd, bun, ji 추출
- **건축물 데이터**: 공공데이터포털 건축물대장 서비스 (무료)

## 🔄 매칭 알고리즘 프로세스

### 1차 필터 (자동)
1. 카카오 API로 주소 → 지번 코드(sigunguCd, bjdongCd, bun, ji) 변환
2. `/getBrExposInfo` → 건물 전체 호별 목록 (동, 호, 층 — `flrNo` 정수 필드 사용)
3. `/getBrExposPubuseAreaInfo` → 전유부 면적 (`exposPubuseGbCd='1'` 필터)
4. (동 번호) → (층 정확일치) → (면적 ±0.1㎡) 순서로 필터링
5. 결과 1개: 호수 확정 출력

### 2차 필터 (재산세 — 동일 면적 후보 복수 시)
- 조건: 1차 필터 후 후보 2개 이상 + 면적이 소수점까지 완전 동일
- 사용자에게 네이버 매물 '세금정보'의 **재산세 본세** 입력 요청 (지방교육세·도시지역분·종부세 제외)
- 후보 호수별로 `/getBrHsprcInfo`를 **개별 호출** (dongNm+hoNm 파라미터로 각 1건씩 — 전체 조회 후 필터 방식 아님)
- 최신 연도 공시가격 추출 → 재산세 계산 → 입력값과 ±1만원 이내 매칭으로 확정
- 재산세 산식: 과세표준 = 공시가격 × 0.45
  - ≤6천만: × 0.1%
  - ≤1.5억: 60,000 + (초과분) × 0.2%
  - ≤3억: 240,000 + (초과분) × 0.3%
  - 3억 초과: 570,000 + (초과분) × 0.4% ← 12억 초과도 동일 브래킷, 종부세 영향 없음

## 📌 주요 버그 수정 이력 (운영 중 발견)
| 증상 | 원인 | 해결 |
|---|---|---|
| 일부 호수만 반환 | 페이지네이션: `numOfRows=1000`이어도 실제 per_page는 ~100 | `len(첫페이지_items)`를 divisor로 사용 |
| 잘못된 층 매칭 ("5" → 15층, 25층 포함) | `str.contains` 사용 | `== f"{floor}층"` 정확일치로 변경 |
| 면적 전부 0 | `/getBrExposPubuseAreaInfo`에 면적 필드 없음 | `/getBrExposInfo`(호목록) + `/getBrExposPubuseAreaInfo`(면적) 듀얼 API 조인 |
| 건물명 필터 후 0건 | 사용자 입력 한글자("샵")과 API 데이터("샾") 불일치 | 건물명 필터 완전 제거 (bun/ji로 필지 특정하면 충분) |
| 2차 확인 느림 | 공시가격 전체 조회(1270건 페이지네이션) 후 2건 필터 | 후보별 개별 API 호출 (dongNm+hoNm 파라미터, numOfRows=10) |
